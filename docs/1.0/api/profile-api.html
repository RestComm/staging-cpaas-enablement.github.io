Profiles

Overview

Profiles allow Communications Providers (Primary Accounts) to enable specific voice and SMS Routing, features, or applications for one or a group of Business Customers (Sub Accounts).

Once a Profile is associated with a Sub Account, the Sub Account will start using that Profile’s configuration.

The following is a high-level diagram that illustrates how the Sub Accounts are associated with the Profiles.

Profile Types

The Profile API allows different Types of Profiles to be provisioned. A Sub Account has always one, and only one, Profile of each Type associated with it. There are currently two Profile Types that can be managed by the Communications Providers:

  • Management Profile, which allows Communications Providers to control access to specific features and applications for each Sub Account.

  • Subscriber Profile, which allows Communications Providers to set their preferable BYOC and Media Session Control settings for each Sub Account’s Calls and SMS traffic.

Management Profile

Only the Administrators of the Primary Account can create and manage Management Profiles. Any other account which is assigned to the Management Profile can read its contents.

In the future, the REST API will allow Communications Providers to manage a wider range of functionalities.

The Profile REST API and Management Profile are only accessible by the Primary Account. Any Administrator of this account can create and manage the Profiles.

Subscriber Profile

Subscriber Profiles enables Communications Providers to configure calls and SMS routing with BYOC as well as Media Session Control for their Business Customers.

Profiles Associations, Default Profiles, and Inheritance

Profiles Associations

Default Profiles

Upon creation, new Sub Account will inherit the Subscriber Profile assigned to the Organization, by the Communications Provider, that contains BYOC and Media Session Control settings. In case the Organization does not have a Subscriber Profile explicitly assigned, it will fallback to inherit the CPaaS Platform Default Subscriber Profile and so will the new Sub Account.

The System Default Management Profile allows access to all Turnkey Applications enabled for the Organization.

The System Default Subscriber/Management Profiles are pre-provisioned in the system, and can’t be modified from the REST API.

  • The System Default Subscriber Profile Sid is "PR2a61d1839442571eb2eb88262064d1f9"

  • The System Default Management Profile Sid is "PR90ec08939663ff0e97a310643c62db18"

Profile Inheritance

The effective/active Profile will be computed by checking in the following order, until a Profile is found:

  • Profile associated with the specific Sub Account.

  • Default Profile Associated with the corresponding organization of the Sub-Account

  • System Default Profile

You can check which are the effective Profiles for a given account and organization by retrieving them through their respective API Endpoints and inspecting the Link Response Headers.In addition, you can use the Profiles REST API endpoint to find out with which accounts and organizations a profile is associated.

This lookup process facilitates an inheritance model where more specific Profiles can be created to override more generic ones.When overriding the System Default Profile, the Organization will be associated with the new Organization Profile, and their Business Customers (Sub Accounts) will inherit the same.Every new Sub Account will inherit the Default Management and Subscriber Profiles until a Communications Provider’s Administrator changes them by associating another Profile to this specific Sub Account.

Resource Schemas Details

The REST API will validate well-known sections of the document against a JSON schema.The rest of the document will not be validated, with the only constraint of a maximum size limitation of 10MB.This means that the REST API consumer may use the Profile document to store/retrieve any kind of arbitrary information as long as the document is a valid JSON document.Here are the various JSON schemas (and subschemas) that are being used per profile type:

  • Resource schema for Management Profile: http://yourorganization.com/api/2012-04-24/Profiles/schemas/rc-management-schema.json

  • Resource Schema for Subscriber Profile: https://yourorganization.com/api/2012-04-24/Profiles/schemas/rc-subscriber-schema.json

    • Resource Schema for multiProvider: https://yourorganization.com/api/2012-04-24/Profiles/schema/rc-multi-provider-schema.json

    • Resource Schema for sessionControl: https://yourorganization.com/api/2012-04-24/Profiles/schema/rc-session-control-schema.json

Access Control

Profile access permissions vary based on the user role and account level.

Management Profile Access Control:

  • The Primary Account Administrator (CP/Administrator) can create and edit Management Profiles while the users with other types of roles (CP/Developer and CP/ProvisioningAgent) can only view them.

  • The Primary Account Administrator (CP/Administrator) can assign/unassign their Management Profiles to Business Customer child Accounts

  • Sub Account users can only read the Management Profile assigned to them

Subscriber Profile Access Control:

  • The Primary Account Administrator (CP/Administrators) can create and edit the Subscriber Profiles while the users with other types of roles (CP/Developer and CP/ProvisioningAgent) can only view.

  • The Primary Account Administrator (CP/Administrator) can assign/unassign their Subscriber Profiles to Business Customer child Accounts

  • Sub Account users have no access to the Subscriber Profiles.

Create Profile

POST /restcomm/2012-04-24/Profiles

Creates a new Subscriber or Management Profile and returns the representation.The body of the request is expected to contain a JSON document compliant to the Profile schema.

Content-Type HTTP header needs to be passed specifying that JSON should be provided along with the schema for the created profile.Below are the values of the Content-Type:

  • Management Profile: application/instance+json;schema="<ENDPOINT>/api/2012-04-24/Profiles/schemas/rc-management-schema.json".

  • Subscriber Profile: application/instance+json;schema="<ENDPOINT>/api/2012-04-24/Profiles/schemas/rc-subscriber-schema.json".

You can check out Resource Schemas Details for more information on the various Profile schemas.

After successful creation the profileSid of the new Profile is returned in the Location response header.

Subscriber and Management Profiles can only be created by the CP Administrator.

Establishing media connection when the remote node is behind NAT (Whitelist of nodes IP addresses, SIP inbound)

This functionality helps you enhance your security levels by whitelisting a range of IP addresses allowed to communicate with CPaaS Infrastructure. The feature is disabled by default. It can be enabled when needed by adding the "mediaRemoteWhitelist" parameter to the "sessionControl" subschema of the Subscriber Profile. You need to specify one or more IP addresses that (and only that) will be allowed to initiate connections under this profile. IP addresses may be present in the form of a single address (like 192.168.1.1) or as a range (like 192.168.2.0/24).

curl -X POST -u 'apiKeyValue:apiKeySecret' \
 https://yourorganization.com/api/2012-04-24/Profiles \
 -H 'Content-Type: application/instance+json;schema="https://yourorganization.com/api/2012-04-24/Profiles/schemas/rc-subscriber-schema.json"' \
 -d '{
    "sessionControl": {
        "mediaRemoteWhitelist": [
            "192.168.1.1", "192.168.2.0/24"
        ]
    }
}'

Enabling DNS Resolution

By default, the usage of DNS resolution for calls is disabled. To enable it, you need to add a "dns_resolve" addition of the "multiProvider.outbound-voice.proxy.uri" Subscriber Profile parameter.

curl -X POST -u 'apiKeyValue:apiKeySecret' \
 https://yourorganization.com/api/2012-04-24/Profiles \
 -H 'Content-Type: application/instance+json;schema="https://yourorganization.com/api/2012-04-24/Profiles/schemas/rc-subscriber-schema.json"' \
 -d '{
    "multiProvider": {
        "outbound-voice": [
            {
                ....
                "proxy": [
                    {
                        "uri": "10.0.0.1:5060;dns_resolve",
                        ....
                    }
                ]
            }
        ],
    },
}'
Activating Traces Generation

This session control extension allows generating Traces for the linked account traffic.

curl -X POST -u 'apiKeyValue:apiKeySecret' \
 https://yourorganization.com/api/2012-04-24/Profiles \
 -H 'Content-Type: application/instance+json;schema="https://yourorganization.com/api/2012-04-24/Profiles/schemas/rc-subscriber-schema.json"' \
 -d '{
    "sessionControl": {
        "traceSdrMode": "enabled"
    }
}'

If not specified, Traces won’t be generated for the account traffic.

Make sure to not override the existing profile. The sessionControl extension should be appended to an existing profile.

New profiles needs to be linked to the Organization or Account in order to enable them (as described in subsection Link/Unlink Profile to Account/Organization)
Manipulation of SIP Fields

This session control extension allows fine-grained control on how DNS resolution shall affect different fields and headers of the outbound SIP requests.

curl -X POST -u 'apiKeyValue:apiKeySecret' \
 https://yourorganization.com/api/2012-04-24/Profiles \
 -H 'Content-Type: application/instance+json;schema="https://yourorganization.com/api/2012-04-24/Profiles/schemas/rc-subscriber-schema.json"' \
 -d '{
    "sessionControl": {
        "resolveRequestUri": "resolve(default)|keep",
        "resolveToHeader" : "resolve(default)|keep"
    }
}'

The resolveRequestUri decides whether the hostname in the SIP Request URI shall be replaced with outbound-voice proxy defined in the MultiProvider extension. If the value is set to keep, no resolution is attempted and the field is left unmodified.

The resolveToHeader decides whether the hostname in the SIP To header shall be replaced with outbound-voice proxy defined in the MultiProvider extension. If the value is set to keep, no resolution is attempted and the field is left unmodified.

If not specified, both extensions default to resolve to maintain backwards compatibility.
Mapping of Billing Headers

The Billing Headers are sent in SIP INVITE / MESSAGE requests and responses as SIP custom headers, and in SMPP messages as TLV fields. "P-Asserted-Service" header is used for identifying which service is originating the traffic (e.g. Visual Designer, 2FA, Number Masking, Auto Attendant etc.) for charging/billing purposes. Extra headers can be assigned per Sub Account. Renaming the Billing Headers is possible by adding a "headerTranslationMap" into a Subscriber Profile. This can be specified separately for Voice and Messaging.

Example for voice:

curl -X POST -u 'apiKeyValue:apiKeySecret' \
 https://yourorganization.com/api/2012-04-24/Profiles \
 -H 'Content-Type: application/instance+json;schema="https://yourorganization.com/api/2012-04-24/Profiles/schemas/rc-subscriber-schema.json"' \
 -d '{
"multiProvider": {
  "outbound-voice":[
    {
      "proxy":[
        {
          "uri": "replacement.com",
          "username": "yourusername",
          "password": "yourpassword",
          "headerTranslationMap": {
            "P-Asserted-Service": "X-SomeCustomHeader"
          },
        },
      ]
    }
  ]
}

}'

Example for messages:

curl -X POST -u 'apiKeyValue:apiKeySecret' \
 https://yourorganization.com/api/2012-04-24/Profiles \
 -H 'Content-Type: application/instance+json;schema="https://yourorganization.com/api/2012-04-24/Profiles/schemas/rc-subscriber-schema.json"' \
 -d '{
    "multiProvider": {
        "outbound-sms": [
            {
                "headerTranslationMap": {
                    "P-Asserted-Service": "X-TLV-COCTET-3000",
                    "X-MSIDN": "X-TLV-COCTET-3001"
                }
            }
        ]
    }

}'

When the headerTranslationMap is applied the "P-Asserted-Service" will be replaced with "X-SomeCustomHeader" for Voice, and "X-TLV-COCTET-3000" for Messaging (if that is a SIP message that reaches a customer).

Translated headers with names that start with "X-TLV-" have a special meaning. Headers with such names will be converted into SMPP TLVs before being sent by the SMPP protocol.

The name has the following format: "X-TLV-<TLV type>-<TLV tag (a decimal value)>". The table below contains a list of possible TLV types.

Provisioning Data Consistency on Updates and Deletes

The Telestax CPaaS platform has a distributed microservice architecture that enables linear scalability during traffic growth and handles a high volume of REST API requests, voice calls, and SMS messages. Each architecture component handles different types of requests.

When modifying the provisioning data via the Management REST APIs, it may take a few moments for the update to reflect the changes in all processing components that handle calls and SMS. Provisioning data modifications are done using the HTTP, PUT, and DELETE methods. The POST method creates records immediately.

This is applicable for the following Management APIs, as well as the Console:

  • Accounts

  • Tags

  • Profiles

  • Clients

  • Incoming Phone Numbers
