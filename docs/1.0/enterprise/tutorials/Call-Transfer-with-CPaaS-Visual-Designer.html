Call Transfer with Visual Designer

Overview

This tutorial will help you create your own Call Transfer Visual Designer application. It is assumed that you have an active CPaaS account. In case you do not, please contact your Communications Provider to request that.

Prerequisites

Before you start, you need to make you have the following prerequisites.

  • API Key/Account Sid credentials to make REST API calls

  • Two registered SIP clients to use for Attended Refer scenario

  • A registered SIP address to use for Unattended Refer scenario

Setting up Your Attended IVR Visual Designer Template

Follow the steps below to setup your Attended IVR Visual Designer application template.

  • Log into your CPaaS account

  • Navigate to Applications → Your Applications

  • Click on the + Add New App button

  • Choose to Create a Visual Designer Application

  • Select the Call Transfer template from the Template Gallery

  • Give a name to your application, the click Create.

Setting Up the Application Parameters

Once the creation of the application is successful, you will be taken to the Visual Designer setup screen, where you can set your application parameters. Fill out the parameters fields and click Save.

Parameter Name Description

accountSID

Enter the Account SID to be used in REST API requests.

apiUsername

Enter your username for accessing the REST API.

For non IAM (Identity Access Management) Migrated organizations, this will be the same as the Account SID, Otherwise, this will be the API Key value.

apiPassword

Enter your password for accessing the REST API.

For non IAM Migrated organizations the value is the Auth token, Otherwise, this will be the API Key secret.

apiDomain

Enter your CPaaS organization domain to access the API. E.g. https://yourcompany.com

applicationSID

This Application SID to be used in REST API requests.

The newly created application has an Application ID associated with it. To copy that, navigate to your application inside the Visual Designer editor. Then copy the Application ID value from within the URL that loaded in your web browser. The URL pattern is as follows, where AP6b0423b3d923453daf8694a321a1a8bc is the Application ID.

https://yourcompany.com/visual-designer/#/designer/`AP6b0423b3d923453daf8694a321a1a8bc

salesClientAttended

Client to forward calls for attended call transfer.

The Sales department Sip client that has been registered as part of the prerequisites.

supportClientAttended

Client to forward calls for attended call transfer.

The Support department Sip client that has been registered as part of the prerequisites.

customerCareSipUnattended

Sip address to forward calls for unattended call transfer.

The customer care department Sip Address that has been registered as part of the prerequisites.

Adding A Department to the Call Transfer Application Flow

Follow the steps below to add another Department to the Call Transfer application flow.

  • Navigate to the Main IVR module and click on the Add mapping button.

  • Select new Digit (4) and select Attended Transfer as target.

  • In Attended Transfer, add another Control verb, and configure it to handle the new Department, Condition should be set to $Digit with value 4. The Action should be assigned selectedDepartment to the SIP client that is registered for the new department.

Do not forget to add the text to the Say verb for the new IVR option.

  • Save the application and give it a try.

Understanding the Application Modules and Workflow

The Call Transfer template has nine modules that represent both the attended and unattended refer scenarios. Lets have a look into each module.

01 - Main IVR Simple IVR Collect step to prompt the caller to press a department digit, and transfer the caller to a relevant flow module.

02.a - Unattended Transfer Module with single step that performs call transfer to a specified SIP address.

02.b - Attended Transfer Module that acts as a starting point for Attended flow dynamically, based on a user digit selection and the combination of Control verb.

For every newly aded department, you need to add another Control verb that will populate the global variable selectedDepartment variable with the corresponding department client.

In the case of this template, the department client is exposed as a template parameter. Thus, we can add more departments to this module.

In order to add more departments, the Control step must be duplicated and: * a new $digit value for evaluation in the "condition" section must be set * selectedDepartment variable must be set to new client department in the "Action" section of the Control verb. Don’t forget to add Digit mapping for the new Digit at "Main IVR" module.

At the end of the module, there is a usage of Hold verb which will let the application proceed to the Dial Attended Client module.

03 - Dial Attended Client

Here we use the global variable selectedDepartment (populated in the last module), in the Dial verb with the combination of a Client Noun. This is how the dynamic and reuse of module is done.

Once the Dial to client occurs, the Before Connect module Collect Attended Client", will be processed for the attended department call.

The Before Connect property is hidden by default. In order to reveal it, please click the Dial noun cogwheel.

04 - Collect Attended Client

This module contains a simple Accept/Reject the transfer call via Collect digit verb.

This is what will be presented to the attended department call.

If 1 is press, the application proceeds to Attended Client Accepts module, if 2 is pressed, Attended Client Rejects module is executed.

05.b - Attended Client Rejects

Simple Hung Up step to terminate the transfer.

05.b - Attended Client Accepts

In this module, we perform the actual merge between the original call, and the attended department which accepted the call. We perform Live call modification REST API request via an External Service verb.

The REST request structure is as follows:

https://$apiDomain/restcomm/2012-04-24/Accounts/$accountSID/Calls/$ParentCallSid.json

The first two parameters (apiDomain, accountSID) are defined in the template parameters list. The last one (ParentCallSid) is autopopulated internally via Visual Designer logic.

Do not change or populate this variable yourself.

The API Request required two more query parameters:

ChildCallMode = StayConnected All call legs will continue to stay connected and process an RCML from an existing URL.+ Url - that will serve an empty refer RCML response, in our case we serve the RCML by providing full URI to another module, "Implicit Call Transfer".

The URI is constructed of 3 parameters:

https://$apiDomain/visual-designer/services/apps/$applicationSID/controller?target=$lcmEmptyReferModuleID

The first two parameters (apiDomain, applicationSID) are defined in template parameters list. The last one (lcmEmptyReferModuleID), is assigned one step before the External Service via a Control step.

The template assumes that the module Ids will not change during application modification, but if so, the module ID that represents the empty refer RCML should be assigned to lcmEmptyReferModuleID.

The module ID could be obtained by hovering over the module tab name.

Implicit Call Transfer

This module is for the External Service usage. It outputs the empty call transfer RCML that the API call for live call modification needs.

General Error

Just a simple generic module for handling errors across the application.
