Prerequisites

Before you get started with the Identity Access Management (IAM) migration process, make sure you have the following prerequisites.

  • You have registered your own CPaaS organization

  • You are using the correct organization domain to access the CPaaS Platform Console UIs and corresponding APIs

All API requests can only be authenticated by your Account SID and AuthToken, unless explicitly stated otherwise.

Enabling Identity Access Management (IAM) for Your Organization

Contact your account manager and set a time frame to enable IAM for your organization. Our Global Support Service team will then get in touch with you and walk you through the process.

Accounts Migration Steps

The IAM migration process contains a couple of steps that apply for both the main Account and Sub Accounts of your CPaaS organization.

When IAM is enabled by our service team, any incoming request will automatically trigger the migration process for the authenticated account.

Once the account migration is completed you will have access to the following:

  • A main account user with CP/Administrator role and corresponding account credentials (email and password) is created. You can use them to access the CPaaS Platform UIs.

  • An API key is generated with your existing AccountSID and AuthToken as credentials. You can use them to access the APIs.

  • The following IAM API endpoints will be available for you to use:

    • Users /Accounts/accountSid/Users

--header 'Accept: application/json' \
-u 'YourAPIKeyValue:YourAPIKeySecret'

The role and password/AuthToken at the account level are deprecated once your Account is migrated to IAM. Modifying those through the Console UI will not take effect on the Users or their API keys.
The Account SID should be used as account identifier when making API requests. Using the email address as an Account identifier may fail, and will be deprecated in the future.

Password Policy

The password you are setting when creating an account must pass the complexity test present in the UI and comply with the following policy:

  • contain at least 8 characters

  • contain at least one lowercase

  • contain at least one digit

Authentication Policy

Every user has up to five password attempts. Each additional incorrect attempt will lock out the user for a small period of time that can increase to maximum 15 minutes.

If you forget your password, and you would like to reset it, please contact our support team.

Enabling IAM for a Business Customer

Communications Providers can trigger IAM migration for their Business Customers (Sub Accounts) by making a request to any API endpoint using the Business Customer’s Sub Account SID and AuthToken. You can retrieve a list of your Business Customers and their SIDs and tokens by making a request to the Accounts API endpoint, or by navigating to the Sub Accounts page in the Console.

After the migration a user will be created with the corresponding role as the migrated Sub Account.

When a Developer Sub Account is migrated, there will be only a BC/Developer user under it. If you would like to grant full permission to the Sub Account, you will need to create a new user with BC/Administrator role.

Retrieving a List of Migrated Business Customers

Communications Providers can verify if a Business Customer is migrated by making a request to the Sub Account Users API endpoint. If a Sub Account has not been migrated you will get a response without any Users.

Sub Accounts that have been migrated will have the BC type assigned to them. Not migrated Sub Accounts will have Full type assigned them.

Example Request

curl -X GET https://yourcompany.com/api/2012-04-24/Accounts  \
--header 'Accept: application/json' \
-u 'YourAPIKeyValue:YourAPIKeySecret'

Creating Business Customer Accounts when IAM Migration is Completed

After your organization’s IAM migration has been completed, all newly created Business Customer accounts will also be IAM enabled but will not have an API key generated automatically for accessing the APIs.

The Business Customer account should issue a request to the API Keys endpoint in order to create a new API key. Once the key is generated, all API requests shall use the API key credentials.

You must use the email and password to create the first API key. But using the email and password to authenticate any other API requests is not recommended and will be deprecated in the future.

Example Request

curl --location --request POST 'https://yourcompany.com/api/2012-04-24/Accounts/AC61148eb9d61281750ccc0e13c6094fe1/Users/USaff6a6dec30047d58d0f08edf5f46f0f/APIKeys' \
--header 'Accept: application/json' \
-u 'YourUsername:YourPassword'
